# Database Integration Guide

## Overview
RapidAid uses **Supabase (hosted Postgres)** for persistent, **shared** storage. Requests submitted
by anyone are visible to everyone — there is no per-browser data. (This replaced the original
IndexedDB implementation on 2026-06-18.)

## Architecture

```
Form Submission → Gemini AI → Supabase (Postgres) → Dashboard Display
     ↓              ↓              ↓                    ↓
RequestForm.jsx → geminiService.js → databaseService.js → DashboardPage.jsx
                                         ↑
                                  supabaseClient.js (shared client)
```

## Connection (`supabaseClient.js`)
Creates one shared Supabase client from env vars:
- `VITE_SUPABASE_URL` — `https://pjjsasukxjafqrgnmzpe.supabase.co`
- `VITE_SUPABASE_ANON_KEY` — the publishable (`sb_publishable_…`) key

Both are safe to ship in the browser; **Row-Level Security (RLS)** is what protects the data.

## Database Service (`databaseService.js`)

A singleton whose method names are unchanged from the old IndexedDB version, so callers
(`geminiService.js`, `DashboardPage.jsx`) did not change.

### Key methods
- `initialize()` — no-op (kept for API compatibility; Supabase needs no per-browser setup)
- `addRequest(data)` — inserts a request. **Whitelists real columns**, so any client-supplied
  `id` / `created_at` / `error` fields are dropped and the database generates `id` (UUID) and
  timestamps. Returns the saved row.
- `getAllRequests()` — all requests, newest first
- `getRequestsByCategory(category)` — filters with `contains` on the `categories[]` array
- `getRequestsByStatus(status)` — filters by status
- `updateRequestStatus(id, status)` — updates status + `updated_at`
- `getStats()` — counts by category/status + recent 5
- `migratePendingToUnclaimed()` — normalizes legacy `pending` → `unclaimed`
- `deleteRequest(id)` / `clearAllRequests()` — exist, but **deletes are blocked by RLS** for the
  public key (see below), so they no-op in the browser.

## Schema

Managed via the Supabase MCP (no committed SQL migration files). Three tables:

- **`requests`** (live):
  `id uuid pk default gen_random_uuid()`, `name text`, `location text not null`,
  `request_text text not null`, `categories text[] not null default '{}'`,
  `status text not null default 'unclaimed'`, `priority text default 'normal'`,
  `latitude/longitude double precision`, `contact_info text`,
  `is_anonymous boolean default false`, `org_id uuid`, `claimed_by_org uuid`
  (both FKs → `organizations`), `created_at/updated_at timestamptz default now()`.
  Indexes: `status`, `created_at desc`, `gin(categories)`.
- **`organizations`** and **`profiles`** — created empty, reserved for the Phase 2 auth/orgs work.

## Row-Level Security (RLS)

RLS is **ON** for all three tables.

- **`requests`** — public policies for `select`, `insert`, and `update` only. **No delete policy**,
  so the anon key cannot delete (a DELETE returns success but removes 0 rows). To clear data during
  development, use the privileged Supabase MCP connection, not the app.
- **`organizations`** / **`profiles`** — RLS on with **no policies** = fully sealed (no anon
  read/write) until Phase 2 introduces authenticated, role-based access.

## Data Flow

### 1. Submission
```javascript
RequestForm → geminiService.categorizeRequest() → databaseService.addRequest()  // → Supabase
```

### 2. AI processing (shape saved to the DB)
```javascript
{
  // id, created_at, updated_at are generated by the database
  name: "John Doe",
  location: "123 Main St, Miami, FL",
  request_text: "Need insulin urgently",
  categories: ["medical", "food"],   // AI-generated; one or more
  status: "unclaimed"
}
```

### 3. Dashboard display
```javascript
DashboardPage → databaseService.getAllRequests() → render
```

## Development Controls (Dashboard)
- **Refresh Data** — reloads requests from Supabase
- **Add Sample Data** — inserts the in-file `mockRequests`
- (**Clear Database** was removed — deletes are RLS-blocked for the public key.)

## Troubleshooting

**Requests not appearing / save errors**
- Check the browser console for Supabase errors and the Network tab.
- Confirm `VITE_SUPABASE_URL` / `VITE_SUPABASE_ANON_KEY` are set in `.env`.

**`new row violates row-level security policy`**
- Expected when writing to `organizations`/`profiles` (sealed) or deleting from `requests`
  (blocked) with the anon key. Use the privileged MCP connection for those.

**AI categorization not saving**
- Check `geminiService` console logs and verify the Gemini key; a failed insert still surfaces in
  the console.

## Production notes
- Data is shared and hosted (Supabase), so it persists across devices and browsers.
- Before going public, RLS is already enabled. Tightening policies per role and moving the Gemini
  key server-side are part of the later auth phase.
